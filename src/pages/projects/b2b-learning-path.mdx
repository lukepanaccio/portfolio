## The problem

commercetools had no structured learning path for B2B modeling—business units, associates, permissions—a high-value but conceptually dense domain for solution architects. Building it meant getting non-obvious modeling constraints *right*: a wrong error shape or an unvalidated assumption teaches architects to build the wrong thing.

## Architecture & why

This is where learning design and engineering meet: I designed the path *and* shipped it, on the eval-driven pipeline.

<MermaidDiagram diagram={`flowchart LR
    A["SME interviews +<br/>claims-validation pass"] --> B["Author for Functional Architect<br/>persona (raw HTTP, no SDK)"]
    B --> C["Verify every API example<br/>against RAML source"]
    C --> D["learning-critique gate (≥ 88/100)"]

- **Decision: ground the path in SME interviews + a claims-validation pass before writing.** *Alternative rejected:* drafting from existing reference docs. *Why:* B2B modeling has non-obvious constraints; unvalidated assumptions (e.g. about certain endpoints and unit-based pricing) had to be removed *before* they became taught material.
- **Decision: author for the Functional Architect persona—raw HTTP, no SDK.** *Alternative rejected:* hand-authoring each page freely. *Why:* running through the eval pipeline enforces accuracy floors and persona alignment consistently, page after page.
- **Decision: verify every API example against the RAML source, not from memory.** *Alternative rejected:* illustrative pseudo-responses. *Why:* a wrong error shape teaches wrong handling. A `403` example was corrected to the *real* `AssociateMissingPermissionError` shape, verified against the RAML—and the `as-associate` / `inheritedAssociates` responses were shape-verified too.
- **Decision: run content-deduplication on the new path before merge.** *Alternative rejected:* skipping it because "it's new content." *Why:* B2B overlaps existing associate/permission reference docs; crosslinking at the paraphrase points cuts RAG collision instead of adding new near-duplicates.

## Evals / validation

Each module is scored by the `learning-critique` three-part hard gate. **Module 2 scored 88/100** for the Functional Architect persona; the refine pass corrected the error-shape example and added shape-verified responses, and dedup added crosslinks at four paraphrase points. Every module carries a scenario-based learning-check assessment.

## Outcome

- **2 modules on a new microsite**—"Model buyer organizations" and "Configure associate access."
- **Module 2: +2,318 lines across 8 pages** (overview, six chapters, learning check), passing the critique gate at **88/100**, with API examples verified against source rather than illustrated from memory.
- A repeatable proof of the learning↔engineering integration: I didn't hand off a spec—I designed the system and shipped it.
