## The problem

The friction between having a musical idea and actually executing it in Ableton is the cost. Menu-diving, hunting selectors, breaking flow. The goal was to drive track/clip/device/MIDI/transport operations in a *live* session by typed or spoken natural language, with **minimal build and a repeatable, low-error setup.** The interesting engineering wasn't the bridge protocol. It was making the integration install reliably and diagnosing why it didn't.

## Architecture & why

To be clear about who did what: the MCP bridge itself (the Remote Script and socket server exposing Ableton's Live API) is **open-source, and I didn't write it.** My contribution is the integration and deployment layer, plus the diagnosis.

- **Decision: integrate a mature open-source MCP server, driven from Claude Desktop.** *Alternative rejected:* Ableton's brand-new official Extensions SDK (released days earlier). *Why:* the SDK is beta, Suite-only, and its run-once-then-stop execution model doesn't fit a *persistent conversational* bridge. The MCP path was mature and worked right then.
- **Decision: pick the Session-view-focused bridge over the alternatives.** *Alternative rejected:* an Arrangement-view-focused server, and heavier 220-322-tool servers. *Why:* the real pain was Session-view friction. The lighter server with the largest community fit that need and beat the heavyweight options that did more than I needed.
- **Decision: macOS native dictation as the speech layer.** *Alternative rejected:* a custom push-to-talk Whisper/Deepgram daemon piping to a headless model. *Why:* zero build, and good enough for "talk → it types into Claude." I'm holding the daemon in reserve, only if true hands-free ever becomes a requirement.
- **Decision: deliver setup as idempotent run-once shell scripts.** *Alternative rejected:* a manual step-by-step runbook, a naive config overwrite, and a bare `uvx` command. *Why:* the scripts merge the Claude config safely (jq plus backup), bake in the *absolute* `uvx` path, and check writability. That removes manual error surface, avoids clobbering other MCP servers, and pre-empts the macOS minimal-PATH silent launch failure.

## Evals / validation

No eval harness. This was integration, so validation was functional and diagnostic, and I won't dress it up as more:

- **Functional smoke tests** against a live session: session-info read, MIDI-track creation, device load, transport/tempo.
- **Log-driven verification:** a read-only diagnostic script parses Ableton's `Log.txt` to confirm which Remote Scripts actually loaded and surface load errors. This is what isolated the failure.
- **Failure handling in the installers:** backup-before-mutate, writability checks, and three explicit outcome branches (installed / found-but-not-writable / drive-not-mounted).

## Outcome

- Working **typed-and-spoken control of a live Ableton session.**
- Manual setup collapsed from a ~7-step runbook to **1 run-once command + 3 irreducible GUI actions** (control-surface select, dictation toggle, verify).
- **2 silent-failure modes pre-empted** in the installer: bare-`uvx` PATH resolution, and config-file clobber.
- **1 non-obvious root cause** found by instrumenting Ableton's own logs. The control surface wouldn't load because the User Library had been relocated to an external drive, and Live 12 loads Remote Scripts only from `<User Library>/Remote Scripts/`, not the default or legacy paths.

> Personal project, framed honestly as **systems integration, tooling, and debugging**. The MCP bridge is open-source and not my work. The idempotent install/fix/diagnostic scripts and the relocated-User-Library diagnosis are mine. A deliberately light piece: no eval harness, thin usage metrics.
