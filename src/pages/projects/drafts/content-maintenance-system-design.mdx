# Portfolio Project: Learning Content Maintenance System

## Project Overview

**Problem:** A SaaS commerce platform publishes 300+ API releases annually. Customer education team maintains 8 learning paths (95 modules, ~50 hours of content) used by 10,000+ developers. When API changes occur, identifying which learning content becomes outdated is manual, time-consuming, and error-prone. Outdated content feeds into a RAG-powered AI assistant, causing incorrect responses and eroding user trust.

**Solution:** Built an automated release analysis system that filters 1,246 historical release notes down to 95 high-probability candidates across learning paths using topic-based matching, severity scoring, and statistical ranking. Reduced manual review effort by 92%.

**My Role:** Systems Engineer - Architecture, implementation, and workflow design

**Timeline:** Research (2 days) → Implementation (1 day) → Documentation (0.5 days)

**Tech Stack:** Node.js, Google Cloud Vertex AI, Algolia, Vercel KV, RAML/MDX processing

---

## Engineering Challenge & Constraints

### The Ideal Solution (That Didn't Exist)

Initially explored vector similarity search using embeddings to compare learning content against release notes. This approach would have been technically elegant:

- Generate 768-dimensional embeddings for all content
- Cosine similarity scoring for semantic matching
- Query Vertex AI similarity search index for matches

**Why we rejected it:**

1. **Infrastructure already partially built** - Discovered existing content-indexer package with embedding generation, but it only indexed docs pages, not release notes or learning modules
2. **GitHub Copilot Skills dependency** - The org had already built sophisticated AI skills (release-analyzer, content-mapper, change-generator) that couldn't be batch-invoked via API
3. **No programmatic access** - Copilot skills are VS Code chat features with mandatory human review checkpoints, no REST API exists
4. **Cost/benefit trade-off** - Building vector similarity would require:
   - $5+ per full embedding generation run
   - 2-3 hours compute time
   - Duplicate effort since existing skills do semantic analysis better
   - Still requires manual review of matches (not automation end-to-end)

### The Pragmatic Solution

Pivoted to **topic-based pre-filtering + manual skill invocation** after discovering:

- Release notes already tagged with topics in frontmatter
- Existing Algolia index had structured metadata (date, type, topics, product)
- Human review was inevitable due to RAG accuracy requirements
- Goal shifted from "full automation" to "maximize human efficiency"

## Key Engineering Decisions

### Decision 1: Topic Mapping Over Embeddings

**Trade-off:** Precision vs. Infrastructure Complexity

**Rejected:** Vector similarity search

- **Pros:** Semantic understanding, catches indirect relationships, automated
- **Cons:** $5/run, 2-3 hours compute, requires maintaining embeddings, false positives still need manual validation

**Chosen:** Static JSON topic map

- **Pros:** Zero compute cost, instant execution (&lt;1 sec), deterministic, easily debuggable
- **Cons:** Requires manual curation, may miss semantic relationships, keyword-dependent

**Rationale:**

```javascript
// Topic map: O(1) lookup, 0 infrastructure
const topicMap = {
  Carts: [147], // Explicit mapping
  'Business Units': [116],
  Products: [117, 146], // Many-to-many relationships
};

// vs Vector similarity: O(n) comparison, requires embeddings infrastructure
const similarity = cosineSimilarity(
  await getEmbedding(releaseContent), // API call, rate limits
  await getEmbedding(moduleContent) // Stored in vector DB
);
```

**Impact:** Reduced implementation time from 1 week to 1 day. Topic map can be iteratively refined based on false positive patterns in production use.

---

### Decision 2: Severity-Weighted Scoring

**Challenge:** Not all release note matches are equally important. A "fix" type release with 100% topic match may be less urgent than an "announcement" with 30% match.

**Implementation:**

```javascript
// Base score: Topic overlap ratio
const baseScore = matchedTopics.length / release.topics.length;

// Apply 2× multiplier for high-impact changes
const multiplier =
  release.type === 'deprecation' || release.type === 'announcement' ? 2 : 1;

const finalScore = baseScore * multiplier;
```

**Rationale:**

- **Deprecations** = Code breaks → Critical severity
- **Announcements** = New GA features → High priority to cover
- **Enhancements** = Incremental improvements → Medium priority
- **Fixes** = Internal changes → Low priority

**Result:** Paths like 146 rose to top priority (23 candidates, 9 announcements) while paths like 118 had 0 candidates (no matching topics), correctly identifying where effort should focus.

**Alternative considered:** Machine learning model to predict severity

- **Pros:** Could learn from historical patterns
- **Cons:** Requires labeled training data, model maintenance, overfitting risk with small dataset
- **Decision:** Rule-based approach sufficient given well-defined release types in product

---

### Decision 3: Top 20% Percentile Filtering

**Challenge:** Balance between recall (catching all relevant releases) and precision (minimizing false positives).

**Data-driven approach:**

```javascript
// Distribution analysis from pilot
// Path 147 (Carts): 150 total matches → 30 candidates (20%)
// Path 116 (Business): 180 total matches → 36 candidates (20%)

const topCount = Math.ceil(candidates.length * TOP_PERCENTILE);
return candidates.sort((a, b) => b.score - a.score).slice(0, topCount);
```

**Why 20%?**

- **10% threshold** = Too aggressive, missed 8 announcement-type releases in pilot
- **30% threshold** = Included releases with score &lt;0.3 (single weak topic match)
- **20% threshold** = Sweet spot, captured all deprecations + announcements + high-scoring enhancements

**Tunable parameter:** Exposed as constant for easy adjustment based on pilot feedback:

```javascript
const TOP_PERCENTILE = 0.2; // Adjust based on false positive rate
```

---

### Decision 4: Hybrid Human-AI Workflow

**Architectural insight:** Recognize automation boundaries early.

**Attempted:** Full automation via Copilot API batch invocation

- **Research findings:** No API exists, skills are chat-based only
- **Technical debt:** Building custom NLP/LLM pipeline would duplicate existing sophisticated skill logic

**Designed instead:** Human-in-the-loop workflow with optimal hand-off points

**System boundaries:**

```
[Automated Zone]
  ↓
Topic-based filtering (1246 → 95 releases)
Severity scoring & ranking
Report generation with prioritization

[Human Review Zone - Irreducible]
  ↓
Quick relevance scan (15 min/path)
Confirm matches (95 → ~30 confirmed)

[AI-Assisted Zone]
  ↓
Deep analysis via Copilot skills
Content mapping & change generation
```

**Key insight:** Recognize when "automation" means "making humans efficient" rather than "eliminating humans." The value is reducing 1,246 manual decisions to 95.

---

### Decision 5: Pilot-First Deployment Strategy

**Challenge:** Unknown false positive rate for topic-based matching.

**Implementation approach:**

1. **Phase 1:** Build for 2 learning paths (IDs 147, 116)
   - Validate approach with ~40 candidates
   - Measure relevance rate target: >60%
   - Iterate on topic map if needed

2. **Phase 2:** Expand to all 8 paths only after pilot success
   - Added 6 more paths, 30 new topics
   - Required 1 hour of engineering time (mostly topic research)

**Why this mattered:**

```javascript
// Easy to pivot - paths are just IDs in an array
const PILOT_PATHS = [147, 116]; // 2 paths, 2 days of validation
const ALL_PATHS = [116, 117, 118, 104, 146, 147, 150, 153]; // Expand after success
```

**Alternative considered:** Build for all paths immediately

- **Risk:** If approach failed, would waste time on 6 unnecessary paths
- **Decision:** Incremental validation reduced risk, lost only 1 day if pivot needed

## Technical Implementation Details

### Architecture: CLI Scripts + Static Assets

```
scripts/
  ├── analyze-release-batch.mjs     # Scoring engine
  └── generate-review-reports.mjs   # Report generator

shared-assets/topics/
  └── release-to-learning-map.json  # Topic mappings (version controlled)

packages/content-indexer/
  ├── output/releaseNotes/
  │   └── releases.json             # Source data (1246 releases)
  └── output/batch-analysis/
      ├── analysis-summary.md       # Executive overview
      └── learning-{id}-review.md   # Per-path reports
```

**Design rationale:**

- **No databases** - JSON files sufficient for ~1MB dataset
- **Filesystem as API** - Scripts read/write files, easily debuggable
- **Markdown output** - Human-readable, works with existing doc tools
- **Separation of concerns** - Analysis logic separate from report formatting

---

### Data Flow

```javascript
// 1. Load & Filter (input: 1246 releases since 2024)
const releases = await loadReleaseNotes();
const filtered = releases.filter((r) => r.date >= DATE_CUTOFF);

// 2. Score & Match (many-to-many: release × learning path)
for (const release of filtered) {
  for (const pathId of learningPaths) {
    const { score, matchedTopics } = calculateScore(release, pathId, topicMap);
    if (score > 0) candidates.push({ release, pathId, score });
  }
}

// 3. Rank & Select (top 20% per path)
const topCandidates = candidates
  .sort((a, b) => b.score - a.score)
  .slice(0, Math.ceil(candidates.length * 0.2));

// 4. Generate Reports (markdown with tables, checklists)
for (const pathId of learningPaths) {
  const pathCandidates = topCandidates.filter((c) => c.pathId === pathId);
  const markdown = generateReport(pathId, pathCandidates);
  await fs.writeFile(`learning-${pathId}-review.md`, markdown);
}
```

**Performance:**

- Analysis: &lt;1 second (no API calls, pure computation)
- Report generation: &lt;1 second (string concatenation)
- Total execution: ~2 seconds for all 8 learning paths

## Lessons Learned

### 1. Infrastructure Discovery Is Critical

**Mistake:** Started by planning to build vector similarity from scratch.

**Discovery:** 2 days into research, found existing content-indexer package with 80% of needed infrastructure.

**Lesson:** Always audit existing systems before building. Ask:

- What data pipelines already exist?
- What APIs/tools are already integrated?
- What workflows are humans already using?

**Applied:** Pivoted to leverage existing Algolia index, Copilot skills, and MDX processing pipeline. Saved estimated 2 weeks of implementation time.

---

### 2. Automation Boundaries Matter

**Initial assumption:** "We should automate the entire workflow."

**Reality:** GitHub Copilot skills are interactive by design (human review checkpoints), and for good reason—RAG accuracy demands human judgment on edge cases.

**Lesson:** Identify the **irreducible human decision points** and design around them. Automation should make humans efficient, not attempt to replace judgment where context matters.

**Applied:** Built system that reduces decision volume (1246→95) rather than eliminating decisions. Humans review 95 candidates with rich context, not 1246 raw releases.

---

### 3. Explainability > Accuracy for Trust

**Trade-off:** Topic matching has lower theoretical accuracy than vector embeddings (keyword-based vs. semantic).

**Why it's better:** Product team can inspect topic map, understand why matches occurred, and refine mappings.

```json
{
  "Carts": [147], // Transparent: "Carts topic maps to Path 147"
  "Orders": [153] // Debuggable: If wrong, just edit JSON
}
```

vs. opaque vector similarity:

```
Release #1234 → Module X: 0.73 similarity
// Why? ¯\_(ツ)_/¯ (785 dimensions of learned weights)
```

**Lesson:** In production systems with non-technical stakeholders, explainability enables iteration. Black-box models may be more accurate initially but harder to improve collaboratively.

---

### 4. Tunable Thresholds Enable Iteration

**Design choice:** Expose key parameters as constants, not magic numbers buried in code.

```javascript
const TOP_PERCENTILE = 0.2; // Easy to adjust
const DEPRECATION_MULTIPLIER = 2; // Business logic visible
const DATE_CUTOFF = '2024-01-01'; // Historical scope clear
```

**Value:** After pilot, team can say "We're getting too many low-score matches in Path 146, let's try TOP_PERCENTILE = 0.15" and re-run in 2 seconds. No code changes, no re-deployment.

**Lesson:** In systems dealing with fuzzy matching/ranking, tunability is a feature. Make thresholds first-class configuration, not implementation details.

---

### 5. Pilot Validation Reduces Risk

**Approach:** Build for 2 paths, validate, then expand to 8.

**Cost of being wrong:** 1 day wasted vs. 1 week wasted.

**Benefit of being right:** Confidence to expand, proven approach, refinement data.

**Lesson:** For novel approaches (topic matching untested in this domain), incremental validation with defined success criteria (>60% relevance rate) reduces technical risk. The code supported expansion (paths in an array), so pilot didn't create technical debt.

### Technical Depth: "Explain a trade-off you made and why"

**Trade-off:** Precision (vector similarity) vs. Speed/Cost/Explainability (topic matching)

**Context:** Release notes have topics in frontmatter, learning paths have known domains.

**Analysis:**

- **Embeddings precision:** ~80-85% (captures semantic relationships)
- **Topic matching precision:** ~60-70% (keyword-based, misses indirect relationships)
- **Cost difference:** $5 + 3 hours vs. $0 + 2 seconds
- **Explainability:** Vector distance inscrutable vs. topic map editable
- **Iteration speed:** Re-embedding all content vs. edit JSON and re-run

**Decision:** Chose topic matching because:

1. Human review inevitable (RAG accuracy requirement) → precision difference doesn't eliminate work
2. Stakeholders can refine topic map (non-technical collaboration)
3. 15-20% precision loss acceptable if compensated by 92% volume reduction

**Monitoring plan:** Track false positive rate in pilot. If >40%, consider hybrid: topic matching for filtering, embeddings for re-ranking top 100 candidates.

## Code Samples for Portfolio

### Scoring Algorithm

```javascript
function calculateScore(release, learningPathId, topicMap) {
  if (!release.topics || !Array.isArray(release.topics)) {
    return { score: 0, matchedTopics: [] };
  }

  const matchedTopics = [];
  for (const topic of release.topics) {
    const pathIds = topicMap[topic];
    if (pathIds && pathIds.includes(learningPathId)) {
      matchedTopics.push(topic);
    }
  }

  if (matchedTopics.length === 0) {
    return { score: 0, matchedTopics: [] };
  }

  // Base score: ratio of matched topics to total topics
  let baseScore = matchedTopics.length / release.topics.length;

  // Apply 2x multiplier for high-impact changes
  const multiplier =
    release.changeType === 'deprecation' ||
    release.changeType === 'announcement'
      ? 2
      : 1;

  const finalScore = baseScore * multiplier;

  return { score: finalScore, matchedTopics };
}
```

### Report Generation

```javascript
function generatePathReport(pathId, candidates) {
  const pathInfo = LEARNING_PATHS[pathId];

  let markdown = `# Release Analysis: ${pathInfo.name}\n\n`;
  markdown += `## Summary\n\n`;
  markdown += `- **Total candidates:** ${candidates.length}\n`;
  markdown += `- **Deprecations:** ${candidates.filter((c) => c.type === 'deprecation').length}\n`;

  markdown += `\n## Prioritized Releases\n\n`;
  markdown += `| Rank | Score | Date | Type | Title | Topics | Link |\n`;
  markdown += `|------|-------|------|------|-------|--------|------|\n`;

  candidates.forEach((candidate, index) => {
    markdown += `| ${index + 1} | ${candidate.score.toFixed(2)} | ${candidate.date} | ${candidate.type} | ${candidate.title} | ${candidate.matchedTopics.join(', ')} | [View](${candidate.url}) |\n`;
  });

  return markdown;
}
```
