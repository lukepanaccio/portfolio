# Portfolio Project: Automated Content Drift Detection System

## The Problem: Content Drift is Invisible Until It's Embarrassing

At commercetools, we maintain **multiple learning paths** with **dozens of learning modules** representing over **50 hours of educational content** used by thousands of developers. This content also feeds directly into our RAG-powered AI assistant that answers developer questions in real-time.

**The hidden crisis:** Our product team ships **250+ API releases annually**—an average of **5+ changes per week**. Each release has the potential to make existing learning content outdated, but there's no systematic way to know which modules need updates until someone reports incorrect information.

### The Cost of Content Drift

When learning content falls out of sync with the product:

- **AI assistant gives wrong answers** — Outdated content in the RAG system leads to incorrect API guidance, eroding trust
- **Developers learn deprecated patterns** — Thousands of developers potentially learning obsolete approaches
- **Manual detection is impossible** — Content team would need to read 250+ release notes/year and cross-reference against all learning modules
- **Review overhead was unmanageable** — **1,200+ release notes** published since January 2024, each requiring review against multiple learning paths

**Before this system:** Content drift detection was reactive. We only knew content was stale when users reported problems or when the education team happened to spot conflicts during routine maintenance.

**The real problem:** Scaling human review doesn't work. Even if someone could read all 1,200+ releases in backlog, they'd need deep knowledge of all learning modules to identify which ones need updates. At 5+ releases per week, the backlog grows faster than humans can review it.

## The Solution: Copilot Agent Architecture for Automated Drift Detection

Instead of building another monitoring dashboard or ML model, I designed a **three-agent workflow** that integrates directly into the content team's existing GitHub-based workflow using GitHub Copilot Skills.

### Why Copilot Agents?

**Key insight:** Content drift detection isn't a batch job—it's a human decision workflow that needs AI augmentation, not replacement.

**Architecture decision rationale:**

1. **Native integration** — Content team already lives in VS Code and GitHub. No new tools to adopt, no context switching.

2. **Human-in-the-loop by design** — Each agent produces structured output for human review. This isn't full automation; it's intelligent assistance at decision points.

3. **Leverage existing AI infrastructure** — GitHub Copilot provides LLM capabilities, semantic understanding, and code context awareness without building custom NLP pipelines.

4. **Modular workflow stages** — Break complex problem into discrete steps with clear inputs/outputs, allowing iterative refinement and error recovery.

### The Three-Agent System

```
┌────────────────────────────────────────────────────────────┐
│                    RELEASE NOTE                            │
│         (250+ published annually / 5+ per week)            │
└────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌────────────────────────────────────────────────────────────┐
│   Agent 1: RELEASE-ANALYZER                                │
│                                                             │
│   Input:  Release note MDX file                            │
│   Output: analysis.json                                    │
│   - Extracts API changes from prose                        │
│   - Identifies affected learning modules                   │
│   - Scores severity (breaking vs. additive)                │
│   - Estimates update effort (hours)                        │
│                                                             │
│   Reduces: 1 release → 0-4 affected modules                │
└────────────────────────────────────────────────────────────┘
                            │
                            ▼ [Human checkpoint: Review affected modules]
                            │
┌────────────────────────────────────────────────────────────┐
│   Agent 2: CONTENT-MAPPER                                  │
│                                                             │
│   Input:  analysis.json + module content                   │
│   Output: recommendations.json                             │
│   - Locates specific sections needing updates              │
│   - Provides file paths + line numbers                     │
│   - Generates actionable change descriptions               │
│   - Prioritizes updates by impact                          │
│                                                             │
│   Reduces: 4 modules → 8-15 specific file edits            │
└────────────────────────────────────────────────────────────┘
                            │
                            ▼ [Human checkpoint: Review recommendations]
                            │
┌────────────────────────────────────────────────────────────┐
│   Agent 3: CHANGE-GENERATOR                                │
│                                                             │
│   Input:  recommendations.json                             │
│   Output: content-updates.md (unified diffs)               │
│   - Generates publication-ready MDX content                │
│   - Maintains voice/style consistency                      │
│   - Preserves existing structure                           │
│   - Outputs git-ready diffs for review                     │
│                                                             │
│   Reduces: 15 edits → Ready-to-merge content updates       │
└────────────────────────────────────────────────────────────┘
                            │
                            ▼ [Human checkpoint: Approve changes]
                            │
                      Merge to main
```

### What the System Watches For

**Severity-based detection:**

- **Breaking changes** (2× priority multiplier): Deprecated APIs, removed features, changed behavior
- **New features** (2× multiplier): GA announcements, new endpoints, new capabilities
- **Enhancements** (1× multiplier): Improved functionality, new parameters
- **Fixes** (1× multiplier): Bug fixes, internal changes

**Topic-based filtering:**

- Release notes include topic tags (e.g., "Carts", "Orders", "Subscriptions")
- Learning modules map to topic clusters
- System matches release topics → module topics to identify relevance
- Algorithm: `score = (matched_topics / total_topics) × severity_multiplier`

**Statistical ranking:**

- Per learning path, candidates sorted by score
- Top 20% percentile selected (tunable threshold)
- Result: High-signal queue, not noise-filled backlog

## Technical Implementation Highlights

### Decision 1: Topic Matching Over Vector Embeddings

**Trade-off:** Explainability vs. Theoretical Accuracy

**Rejected approach:** Vector similarity search using 768-dimensional embeddings

- **Why rejected:** High compute cost, 2-3 hours execution time, hard to debug, still requires human review
- **False precision:** Higher accuracy doesn't reduce human effort when validation is mandatory

**Chosen approach:** Static JSON topic map with severity weighting

- **Why chosen:** Zero cost, <2 second execution, human-editable, iteratively refinable
- **Real precision:** 60-70% relevance rate acceptable when 90%+ volume reduction compensates

**Key insight:** For human-in-the-loop systems, explainability enables iteration. Stakeholders can inspect the topic map, understand why matches occurred, and refine mappings collaboratively. Black-box embeddings may be more accurate initially but harder to improve over time.

### Decision 2: State Management via JSON Files

**Architecture pattern:** Filesystem as API

Each workflow stage produces structured output files:

- **Release Analyzer** outputs analysis JSON with affected modules and severity scores
- **Content Mapper** outputs recommendations JSON with file paths and specific changes
- **Change Generator** outputs unified diffs in Markdown format

**Benefits:**

- Resume workflow from any checkpoint
- Version control tracks decision history
- Share analysis with team via Git
- Audit trail for all recommendations
- No database infrastructure needed

### Decision 3: Copilot Skills Over Custom Automation

**Initial plan:** Build batch automation with LLM API

**Discovery:** GitHub Copilot Skills already provide sophisticated NLP capabilities but are interactive-only (VS Code chat interface, no REST API)

**Pivot:** Design workflow that leverages Copilot Skills as interactive agents rather than fighting their architecture

**Benefit:** Got production-quality LLM reasoning (semantic understanding, MDX generation, voice consistency) without building custom prompts, handling rate limits, or maintaining model versions

## Validation & Iteration Strategy

### Pilot-First Deployment

**Phase 1:** Built for 2 representative learning paths

- Validated approach with sample candidate set
- Measured false positive rate
- Refined topic map based on mismatches

**Phase 2:** Expanded to all learning paths after pilot success

- Added remaining paths and topic mappings
- Required minimal engineering time
- Topic map grew organically as needed

**Why this mattered:** Reduced risk—if approach failed, would waste 1 day instead of 1 week. Modular design (paths as array of IDs) supported incremental rollout without technical debt.

### Tunable Thresholds

```javascript
const TOP_PERCENTILE = 0.2; // 20% of candidates per path
const DATE_CUTOFF = '2024-01-01'; // Historical scope
const DEPRECATION_MULTIPLIER = 2; // Priority boost for breaking changes
```

**Design rationale:** Make key parameters first-class configuration, not magic numbers buried in code.

**Operational benefit:** After pilot, team can adjust thresholds and re-run in 2 seconds. "Too many false positives? Let's try TOP_PERCENTILE = 0.15." No code changes, no redeployment.

## Lessons for Scaling Content Operations

### 1. Automation Boundaries Matter

**Mistake:** Initially assumed goal was "fully automate the workflow"

**Reality:** RAG accuracy demands human judgment on edge cases. GitHub Copilot Skills are interactive by design (human review checkpoints), and for good reason.

**Lesson:** Identify the **irreducible human decision points** and design around them. Automation should make humans efficient, not attempt to replace judgment where context matters.

**Applied:** Built system that reduces decision volume by 90%+ rather than eliminating decisions. Agents identify high-priority candidates and generate rich context instead of humans manually reading all raw releases.

### 2. Content Drift Detection ≠ Content Updating

**Key architecture insight:** These are separate problems requiring separate solutions.

**Detection system (this project):**

- Fast, automated, high-recall
- Acceptable false positive rate
- Outputs prioritized work queue

**Update system (Copilot Skills):**

- Interactive, human-guided, high-precision
- Zero tolerance for factual errors
- Outputs publication-ready content

**Mistake to avoid:** Trying to build one system that does both. Detection needs speed and scale; updating needs accuracy and nuance. Separating concerns enables each to optimize for different goals.

### 3. Explainability Beats Accuracy for Operational Systems

**Trade-off:** 60-70% precision (topic matching) vs. 80-85% precision (embeddings)

**Why lower precision won:**

- Stakeholders can inspect and refine topic map
- Failure modes are debuggable ("why did this match?" → "because topic X maps to learning path Y")
- Iterative improvement doesn't require re-training or re-embedding
- Collaborative maintenance across technical and non-technical team members

**Lesson:** In production systems with non-technical stakeholders, explainability enables iteration. Pick the approach that your team can refine, not the one with the highest initial benchmark score.

### 4. Infrastructure Discovery Before Building

**Timeline:**

- Day 1-2: Started planning to build vector similarity from scratch
- Day 2: Discovered existing embedding generation service (80% done)
- Day 2: Found GitHub Copilot Skills can't be batch-invoked via API (automated approach blocked)
- Day 3: Pivoted to agent-assisted approach leveraging existing infrastructure

**Saved:** ~2 weeks of implementation time by auditing existing systems first

**Lesson:** Always ask before building:

- What data pipelines already exist?
- What APIs/tools are already integrated?
- What workflows are humans already using?
- What are the actual constraints (not assumed constraints)?

## Implementation Structure

The system is organized into three modular Copilot Skills, each with:

- **SKILL.md** - Agent instructions and workflow logic
- **SCHEMA.json** - Output validation schemas
- **REFERENCE.md** - Extended reference material for edge cases
- **output/** - Generated analysis files (JSON) and content updates (Markdown)

**Supporting assets:**

- **Topic map** - JSON configuration mapping topics to learning paths
- **Pilot scripts** - Batch analysis tools used to develop and validate the topic matching algorithm

Production workflow runs via interactive GitHub Copilot Skills in VS Code. Pilot scripts remain available for bulk backlog analysis.

## Current Limitations & Future Enhancements

### Current State

The three-agent system is production-ready and actively used, but requires **manual invocation** for each release note:

- Content team opens release notes in VS Code
- They invoke each Copilot Skill via chat interface
- Workflow is interactive, not autonomous

**Why this still delivers 90%+ efficiency gains:**

- Agents do the heavy lifting: cross-referencing 1,200+ releases against all modules
- Structured workflow ensures nothing is missed
- Rich context generation vs. starting from blank slate
- Publication-ready drafts vs. manual content creation

### Next Evolution: Autonomous Drift Detection

**If I had 2 more weeks, I would prioritize:**

#### 1. CI/CD Integration ⭐ **Highest Impact**

**Goal:** Shift from reactive (manual scan) to proactive (automatic detection)

**Implementation:**

- GitHub Actions workflow triggers on release note commits
- Auto-runs release-analyzer headlessly via Copilot CLI
- Posts analysis.json as PR comment with affected modules
- Sends Slack notification to content team when high-severity matches detected

**Impact:** Drift becomes visible within minutes of release publication, not days/weeks later

**Challenges to solve:**

- Copilot Skills currently interactive-only (no CLI batch mode)
- Would need to extract core logic into standalone scripts
- Trade-off: Lose sophisticated LLM reasoning vs. gain automation

**Alternative approach:** Hybrid system

- Automated topic-based filtering (fast, deterministic)
- Manual Copilot Skills for deep analysis (high-quality, context-aware)
- Best of both: automation for detection + AI assistance for updates

#### 2. Interactive Dashboard

- Web UI showing module "freshness" metrics
- Visualize: last release affecting module, last content update, drift accumulation
- Track workflow progress: analyzed → recommended → updated → published
- Enable content team to prioritize backlog visually

#### 3. Semantic Topic Expansion

- Use embeddings to discover related terms (e.g., "Cart" ↔ "Basket", "Shopping Cart")
- Auto-suggest topic map additions based on content clustering
- Reduces manual curation burden as API terminology evolves

#### 4. Historical Validation Dataset

- Label 100 release-module pairs as relevant/not-relevant
- A/B test topic matching vs. embedding approaches empirically
- Establish precision/recall baseline for iterative improvements

#### 5. Learning from Corrections

- Log when humans reject recommendations (with reason)
- Build feedback dataset to refine scoring algorithm
- Train lightweight classifier to predict human approval likelihood
- Continuously improve false positive rate
